## syncronized的原理

将同步代码块转化为指令后

* 加锁是monitorenter指令，对应的C++的函数就是`ObjectMonitor::enter(traps)`，作用是通过CAS操作将指定对象的内置锁的owner属性（标识拥有当前锁的线程）改为当前线程，成功则代表无线程持有对象内置锁或者当前线程持有对象内置锁，失败则代表其他线程持有了这个对象内置锁，线程将会挂起。

* 解锁是monitorexit指令`ObjectMonitor::exit(not_suspended, traps)`，作用是通过CAS操作将内置锁的owner属性从当前线程改为null。释放锁以后唤醒_cxq\entryList里挂起的线程。

通常所说的对象的内置锁，是对象头Mark Word中的重量级锁指针指向的monitor对象，该对象是在HotSpot底层C++语言编写的(openjdk里面看)

```cpp
//结构体如下
ObjectMonitor::ObjectMonitor() {  
  _header       = NULL;  
  _count       = 0;  
  _waiters      = 0,  
  _recursions   = 0;       //线程的重入次数。syncronized是一个可重入锁
  _object       = NULL;  
  _owner        = NULL;    //标识拥有该monitor的线程
  _WaitSet      = NULL;    //等待线程组成的双向循环链表，_WaitSet是第一个节点
  _WaitSetLock  = 0 ;  
  _Responsible  = NULL ;  
  _succ         = NULL ;  
  _cxq          = NULL ;    //多线程竞争锁进入时的单向链表。线程挂起，因此syncronized是一个重量级锁
  FreeNext      = NULL ;  
  _EntryList    = NULL ;    //_owner从该双向循环链表中唤醒线程结点，_EntryList是第一个节点
  _SpinFreq     = 0 ;  
  _SpinClock    = 0 ;  
  OwnerIsThread = 0 ;  
}  
```

## syncronized是悲观锁吗

### 悲观锁与乐观锁

锁的一种宏观分类方式是**悲观锁**和**乐观锁**

悲观锁思想是假设最坏的情况，认为共享资源每次被访问的时候就会出现问题(比如共享数据被修改)，所以每次在获取资源操作的时候都会上锁，这样其他线程想拿到这个资源就会阻塞直到锁被上一个持有者释放。而Java线程是通过内核线程实现的，线程调度需要操作系统的支持，导致用户态到核心态转换。

乐观锁思想不去考虑共享资源被访问的时候会不会出现问题，线程可以不停地去尝试访问，只是在提交修改的时候去验证对应的资源（也就是数据）是否被其它线程修改了（具体方法可以使用版本号机制或 CAS 算法）。

可以理解为悲观锁代表总是要阻塞线程的互斥同步方案，乐观锁代表不需要阻塞线程的非阻塞同步方案。

### syncronized的实现机制不全是悲观锁

 Java 中`synchronized`和`ReentrantLock`等独占锁就是典型的悲观锁。

但是synchonize作为Java推崇的一个同步方案，在不断的进行优化，JDK 6 发布的内部synchonize也存在乐观锁的机制，以避免悲观锁使线程频繁地切入核心态之中带来的性能问题。

例如，synchonize具有锁升级（锁膨胀）机制：无锁→**偏向锁** → **轻量级锁** → **重量级锁**。

每一次锁竞争的过程依赖**CAS**算法来将锁的持有者信息修改为当前线程。

每一次升级的过程依赖**CAS**算法来修改对象内置锁的指向。

其中轻量级锁和重量级可以是**自旋锁**，通过短时间的忙等，换取线程在用户态和内核态之间切换的开销。

即时编译的逃逸分析，可以实现**缩消除**或**锁粗化**

## 乐观锁的实现原理是什么

乐观锁是一个宏观的概念。乐观的并发策略不管线程冲突风险，先进行操作，如果没有其他线程争用共享数据，操作成功；如果共享的数据的确被争用，产生了冲突，那再进行其他的补偿措施。采用乐观锁的资源访问同一时间虽然有多个线程操作，但结果只有一条线程能成功，因为不再需要把线程阻塞挂起，因此也被称为非阻塞同步。

不同语言对乐观锁有不同的落地实现，Java采用的是比较并交换（Compare-and-Swap，CAS）；

* 先比较一下值是否与预期值一致，如果一致，交换，返回true
* 先比较一下值是否与预期值一致，如果不一致，不交换，返回false

CAS同步必须要求操作和冲突检测这两个步骤具备原子性，只能靠硬件保证某些从语义上看起来需要多次操作的行为可以只通过一条处理器指令就能完成。在单核CPU上只需要依赖一个指令实现，假如是多核CPU，那么多个核之间还需要额外依赖系统底层的Lock来保证CAS操作的原子性。

Unsafe 是位于 sun.misc 包下的一个类，主要提供一些用于执行低级别、不安全操作的方法，被称为魔法类，其中可以调用Native方法完成CAS操作。如果是在Java语言层面上可以利用J.U.C包里面的整数原子类进行CAS操作。
